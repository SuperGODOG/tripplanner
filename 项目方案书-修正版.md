## 一、项目概述

旅行规划需用户在多个平台间手动切换查询景点、天气、酒店信息，流程割裂且无法自动整合。本系统基于 LangGraph + HelloAgents 框架，构建多 Agent 图编排引擎，端到端生成结构化旅行计划。

**一句话**：4 个 Agent 通过 LangGraph 图编排 + MCP 调高德地图，零人工介入完成"输入城市→输出完整行程"。

## 二、面试引导策略（三层递进）

### 2.1 第一层：抛概念埋钩子

| 你主动说 | 埋的钩子 |
|---------|---------|
| "我们的 Agent 基于 LangGraph 图编排——4 个 Agent 作为图节点，通过状态转移信号触发。" | ▶ 为什么用 LangGraph 而不用手写循环？ ▶ 状态转移信号具体是什么？ |
| "工具链分两层：MCP 远程工具和本地自定义 Tool，注册到同一个 ToolRegistry。" | ▶ 为什么要两层？只用 MCP 不行吗？ ▶ 接口统一怎么实现？ |
| "记忆自己实现了领域分类→权重赋分→整合去重，不是直接用 MemoryTool。" | ▶ 权重怎么算的？ ▶ 跨会话怎么保证记忆不冲突？ |

### 2.2 第二层：展开细节

| 追问 | 回答 |
|------|------|
| "为什么用 LangGraph 不用手写 ReAct？" | 手写 ReAct 适合单 Agent 的 Thought-Action-Observation 循环。但 4 个 Agent 有明确上下游——先景点再天气再酒店再规划——这是**多节点编排**问题。LangGraph 把节点间依赖声明为 Edge，每个 Agent 只管自己节点，编排逻辑从业务代码中分离出来，可持久化（Checkpoint）、可可视化。手写 while + if/else 也能串联，但状态管理、暂停恢复、流程可视化都要自己造轮子。 |
| "状态转移怎么实现？" | 每个 Agent 返回 status 标记（success/failed/skip），LangGraph 的 conditional edge 据此路由。景点搜失败→天气继续执行→规划器 prompt 里标注"景点不可用"。不抛异常，降级而非崩溃。 |
| "MCP 和本地 Tool 接口怎么统一？" | MCPTool 继承自 HelloAgents 的 Tool 基类，跟本地 ValidateTool/FallbackTool 同父类。ToolRegistry 不区分远程/本地——注册进去都一样。共用同一个 MCPTool 实例避免重复建连（每次 ~500ms 握手）。 |
| "记忆权重策略具体怎么设计？" | 每条记忆计算：领域权重 × 衰减权重 × 交互修正 × 冲突惩罚。领域权重（天气 1x、景点 2x、酒店 1.5x），衰减是指数衰减(time)，交互修正 = 1.0 + 修改次数×0.5 + 确认次数×0.2，冲突惩罚 = 0.5（新记忆与历史偏好冲突时）。Top-N 保留，末尾打整合标记去重。 |

### 2.3 第三层：工程取舍

| 追问 | 回答 |
|------|------|
| "为什么不全用 MCP？" | MCP 适合调外部服务。数据校验是纯计算逻辑，本地 Tool 更轻——不需要经过 MCP 协议层（进程通信、JSON-RPC 封装），直接在 Agent 进程内执行。原则：外部数据走 MCP，内部逻辑走本地 Tool。 |
| "4 个 Agent 共用 MCP 会冲突吗？" | 不会。一个 MCPTool 实例同时注册给多个 Agent，框架带 session_id 区分来源。故意共用——避免重复建 MCP 连接。 |
| "权重策略怎么防过拟合？" | 冲突检测机制：新记忆跟历史偏好冲突时冲突惩罚 = 0.5，需多次确认才能提升。防止用户偶然说一次"吃辣"就永久改变推荐。 |

## 三、技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 编排引擎 | LangGraph StateGraph | 声明式图编排，支持条件边 + Checkpoint 持久化 |
| Agent 基类 | HelloAgents SimpleAgent | ReAct 循环开箱即用 |

| 本地自定义工具 | HelloAgents Tool 抽象层 | 继承 Tool + 注册 ToolRegistry |
| 记忆管理 | 自定义记忆模块 | 领域分类 + 权重赋分 + 整合去重 |
| LLM | HelloAgentsLLM (DeepSeek) | 已有 API Key |

## 四、架构设计（5 个核心模块）

### 4.1 多 Agent 图编排（第 4-5 层）

```
架构层级映射：
  第 5 层 — 多智能体编排：4 个 Agent 节点 + Edge 流转
  第 4 层 — 图编排框架：LangGraph StateGraph（状态机 + Checkpoint）
  第 3 层 — 框架封装：HelloAgents SimpleAgent / Tool / ToolRegistry
  第 2 层 — Agent 内循环：每个 Agent 内部的 ReAct while 循环
  第 1 层 — 裸 LLM 调用：HelloAgentsLLM.invoke()
```

| 节点 | 职责 | 调用工具 |
|------|------|---------|
| Agent 1 景点搜索 | 根据城市偏好搜索 | MCP: maps_text_search |
| Agent 2 天气查询 | 查询逐日天气 | MCP: maps_weather |
| Agent 3 酒店推荐 | 搜索预算内酒店 | MCP: maps_text_search |
| Agent 4 行程规划 | 整合前三者生成计划 | 无外部工具（纯 LLM 推理） |

每个节点完成后返回 status 标记。LangGraph conditional edge 根据标记路由（图级条件路由）。失败不中断流程，下游降级执行。

### 4.2 双层工具链

| 层级 | 组件 | 职责 |
|------|------|------|
| 远程 | MCPTool(amap) | 景点搜索、天气查询 |
| 本地 | FormatTool | MCP 返回 JSON → 结构化文本（非 LLM，纯文本处理） |
| 本地 | ValidateTool | 检查数据完整性 → 补默认值 |
| 本地 | FallbackTool | 异常时生成降级模板 |
| 注册 | ToolRegistry | 统一查询，Agent 不区分远程/本地 |

> FormatTool 是非 LLM 的纯文本处理工具：解析 MCP 返回的原始 JSON，提取关键字段，输出结构化文本。Agent 拿到的始终是干净的格式化结果，LLM 上下文不被格式化任务挤占。

### 4.3 错误恢复策略（两层协同）

**图级 — Conditional Routing（LangGraph 第 4 层）：**

| 场景 | 行为 | 效果 |
|------|------|------|
| 搜索超时 | 天气/酒店正常执行，规划节点跳过景点 | 仍获得天气+酒店+基本行程 |
| 天气无数据 | 景点/酒店正常，规划给通用穿衣建议 | 完整行程+天气缺失标注 |
| 酒店搜不到 | 景点/天气正常，规划按预算推荐住宿 | 景点+行程完整 |

**Agent 级 — Error-as-Observation（ReAct 第 2 层）：**

| 坏做法 | 好做法 |
|--------|--------|
| 工具抛异常 → Agent 崩溃 → 用户看到 500 | 工具返回错误文本 → Agent 在 Observation 中看到 → ReAct 自主决定重试/降级 |

> 关键区分：图级做"节点间路由"（这个节点失败了，下游怎么走），Agent 级做"调用内决策"（这次调用失败了，重试还是换参数）。两层各司其职。

### 4.4 记忆模块流水线

1. 领域分类：对话内容分为 5 类（天气/景点/酒店/偏好/通用）
2. 标签标注：提取关键信息（城市、价格偏好、饮食偏好）
3. 权重计算：
   - 领域权重：天气 1x / 景点 2x / 酒店 1.5x
   - 衰减权重：指数衰减(time)
   - 交互修正：1.0 + 修改次数×0.5 + 确认次数×0.2
   - 冲突惩罚：与历史偏好冲突时 = 0.5，否则 = 1.0
   - **最终权重 = 领域权重 × 衰减权重 × 交互修正 × 冲突惩罚**
4. 降序排列：按权重保留 Top-N
5. 去重标记：末尾追加整合标记，防重复写入
6. 持久化：写入存储 + 更新 metadata

### 4.5 架构层级总览

```
    ┌──────────────────────────────────┐
    │ 第 5 层 多智能体编排              │  ← TripPlannerGraph（4 Agent 节点）
    │   - 节点间状态路由                │
    │   - Conditional Routing（图级）    │
    ├──────────────────────────────────┤
    │ 第 4 层 图编排框架                │  ← LangGraph StateGraph
    │   - Node / Edge / Checkpoint      │
    ├──────────────────────────────────┤
    │ 第 3 层 框架封装                  │  ← HelloAgents
    │   - SimpleAgent / ToolRegistry    │
    ├──────────────────────────────────┤
    │ 第 2 层 Agent 内循环              │  ← ReAct while 循环
    │   - Error-as-Observation（Agent级）│
    ├──────────────────────────────────┤
    │ 第 1 层 裸 LLM 调用               │  ← HelloAgentsLLM.invoke()
    └──────────────────────────────────┘
```

## 七、面试应答模板

### 7.1 第一层：开场

我做了个多智能体旅行规划系统。技术栈是 LangGraph + HelloAgents + MCP。4 个 Agent 作为图节点按流水线执行，通过 LangGraph conditional edge 控制状态转移。工具链分两层——MCP 调高德地图，本地 Tool 做校验降级，统一注册。记忆自己实现了领域分类→权重赋分→整合去重。

### 7.2 第二层：细节

**Q: 为什么用 LangGraph 不用手写 ReAct？**

手写 ReAct 的 while 循环适合单 Agent 的 Thought-Action-Observation 流程。但 4 个 Agent 有明确上下游依赖，这属于多节点编排问题。LangGraph 把节点间依赖声明为 Edge，提供 Checkpoint 持久化和 conditional edge 条件路由。手写 if/else 也能串联，但状态管理、暂停恢复、流程可视化都要自己造。

**Q: MCP 和本地 Tool 接口怎么统一？**

MCPTool 继承自 Tool 基类，跟本地工具同父类。ToolRegistry 不区分远程/本地。

**Q: 记忆权重策略具体怎么设计？**

领域权重 × 衰减权重 × 交互修正 × 冲突惩罚。Top-N 保留，冲突检测防过拟合。

### 7.3 第三层：取舍

**Q: 为什么不全用 MCP？**

MCP 适合调外部服务。校验是纯计算逻辑，本地 Tool 不需要经过 MCP 协议层开销。

**Q: 4 个 Agent 共用 MCP 会冲突吗？**

不会。一个实例注册给多个 Agent，框架带 session_id 区分来源。共用避免重复建连。

**Q: 权重策略怎么防过拟合？**

冲突检测：新记忆与历史偏好冲突时冲突惩罚 = 0.5，需多次确认才能提升。
