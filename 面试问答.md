# TripPlanner 面试问答

---

## 项目介绍（30 秒，埋钩子引导追问）

> "我做了一个多智能体旅行规划系统，端到端从输入城市到输出结构化行程，中间不需要人工介入。
>
> 编排层基于 LangGraph 自己设计了一个 5 Node 的 StateGraph——4 个 Agent 加一个记忆加载节点，图级 Conditional Routing 和 Agent 级 Error-as-Observation 两层容错各管各的，一套流程三种降级策略，任何一个数据源挂了都不影响最终输出。
>
> 工具链没有把 MCP 调用的格式化和校验写成独立 Tool——做了个 Wrapper 把三层处理封在一起，Agent 只看到一个接口，后两层纯 Python 不消耗 LLM token。FallbackTool 在三个数据源全挂的时候还能兜底。
>
> 记忆这边自研了一套五因子权重公式——领域权重、时间衰减、交互修正、频率加成、异常惩罚五个维度乘起来。异常检测分两轨——数值型价格走 IQR 统计分布，分类型偏好走频率比。八维用户画像冷启动五次才生效，偶发行为污染不了长期偏好。"

埋了五个钩子，每个后面都有对应回答：编排容错 → 话题 1，Wrapper 设计 → 话题 2，五因子权重 → 话题 3，双轨检测 → 话题 6，Fallback 兜底 → 话题 4。

## 1. LangGraph 图编排

### 第一层（抛钩子）

> "我们的 4 个 Agent 通过 LangGraph StateGraph 编排——5 个 Node 按 Edge 串行，每个 Node 失败不抛异常，写 error_log 后继续。Checkpoint 机制让中断后能从断点恢复。"

### 第二层（业务流程 + 异常处理）

**Q: 为什么用 LangGraph 不用手写 while 循环？**

手写 while 能串联 Agent，但三件事做不到：

1. **声明式图定义**：加 Node 只需 add_node + add_edge，不改现有代码。我们加 memory_node 只加了一个节点和一条边。
2. **Checkpoint 持久化**：每步 State 自动快照。服务器重启后用相同 thread_id 继续，已完成的 Node 不重跑。
3. **Conditional Edge**：当前是 linear，但预留了 conditional routing——景点失败→跳过景点继续天气。

**Q: 5 个 Node 分别做什么？**

| Node | LLM | MCP | 职责 | 失败行为 |
|------|-----|-----|------|---------|
| attraction | ✅ | ✅ | 搜索景点 | 写 error_log，下游继续 |
| weather | ✅ | ✅ | 查询天气 | 同上，给通用穿衣建议 |
| hotel | ✅ | ✅ | 搜索酒店 | 同上，按预算推荐 |
| memory | ❌ | ❌ | 加载用户画像 | 纯本地，基本不会失败 |
| planner | ✅ | ❌ | 整合生成 JSON | 兜底 FallbackTool |

**Q: 图级和 Agent 级错误处理有什么区别？**

图级 Conditional Routing（第 4 层）：Node 返回 status，Edge 决定下一跳。确定性逻辑，不消耗 token。
Agent 级 Error-as-Observation（第 2 层）：工具失败返回错误文本，LLM 自主决定重试/降级。语义理解，消耗 token。

### 第三层（衍生到生产）

**Q: 如果生产环境有 100 个并发请求，LangGraph 怎么处理？**

每个请求独立的 thread_id，LangGraph 的 Checkpoint 按 thread_id 隔离。当前是内存存储，生产应换 PostgreSQL/SQLite 持久化。并发瓶颈不在 LangGraph 而在 LLM API 的 rate limit——需要请求队列 + 重试机制。

**Q: Conditional Edge 没实现，线性图够用吗？**

当前场景够（4 个 Agent 天然串行依赖）。但如果后续加"用户实时修改偏好→重新规划"的交互，conditional edge 就必需了——它能做"用户修改了酒店预算→只重跑 hotel + planner，不动 attraction 和 weather"。

**Q: 线性图够用吗？如何支持 Agent 间反馈？**

当前 5 Node 线性流对单次规划够用，但 LangGraph 的 conditional edge 可以做更进阶的事——Agent 间反馈环。比如：Planner 发现酒店不匹配景点位置或预算，可以通过 conditional edge 回到 hotel Agent 重新搜索：

```python
graph.add_conditional_edges("planner", check_status, {
    "retry_hotel": "hotel",
    "done": END
})
```

只需改一行代码。这是图结构（状态机）相对于传统顺序调用的核心优势——天然支持反馈环，传统 if/else 串联做不到这种"后置节点反推前置节点"的能力。

## 2. MCP 协议与工具架构

### 第一层（抛钩子）

> "工具链分两层——MCP 调高德地图，本地 Tool 做数据处理。设计上我们没用独立 Tool 串联，而是通过 AmapToolWrapper 包装——Agent 只看到一个 Tool，调一次拿到干净结果。"

### 第二层（业务流程 + 异常处理）

**Q: 为什么用 Wrapper 而不是独立 Tool？**

如果 Format/Validate 是独立 Tool，Agent 需要 3 轮 ReAct 调三个 Tool。问题：
1. Agent（LLM）被迫理解数据处理流水线——它不该知道"调完 MCP 还要调 Format"
2. 原始 JSON 上千字符塞进 LLM 上下文，挤占推理窗口

Wrapper 内部 MCP→Format→Validate 三步。Format 和 Validate 是纯 Python 字符串处理——不调 LLM，确定性输出。

**Q: MCPTool 底层怎么工作的？**

```python
MCPTool(server_command=["uvx", "amap-mcp-server"], auto_expand=True)
```

1. 启动子进程运行 `uvx amap-mcp-server`
2. 父进程通过 stdio 用 JSON-RPC 通信
3. auto_expand：子进程返回的工具列表自动注册（maps_text_search, maps_weather 等 16 个）
4. 每次 mcp_tool.run() 序列化参数→写 stdin→子进程调 HTTP API→读 stdout

**Q: 4 个 Agent 共用 MCPTool 会冲突吗？**

不会。共享一个实例避免重复建子进程（每次 ~500ms 握手）。框架带 session_id 区分调用来源。

**Q: FallbackTool 什么时候触发？**

当三个数据源（景点/天气/酒店）全部失败时，Planner 无法整合有效数据 → FallbackTool 生成降级模板（"请到达后咨询当地旅游信息中心"），而不是抛 500。

### 第三层（衍生到生产）

**Q: 如果高德 API 挂了但不是全挂（比如天气可用但景点挂了），怎么处理？**

当前是最终节点的降级标注："⚠️ 景点数据不可用，行程中景点为通用推荐"。更优方案是给每个数据源独立的 Fallback——天气挂了用缓存的历史均值，景点挂了用热门景点知识库。

**Q: MCP 协议在实际生产中的坑？**

1. stdio transport 不适合分布式——多台机器没法共享子进程。生产应换 SSE/HTTP transport。
2. 工具发现（auto_expand）每次启动都重新握手——生产应加工具列表缓存。
3. MCP Server 自身无认证——生产需要 API Gateway 做鉴权。

## 3. 记忆系统与用户画像

### 第一层（抛钩子）

> "记忆模块不是简单存/取。五因子权重公式包括频率加成和基于 IQR 的异常检测。用户的长期偏好会自动浮上来，偶然行为会被自动降权。"

### 第二层（业务流程 + 异常处理）

**Q: Alex 案例是怎么回事？**

Alex 5 次出行选经济型酒店（300-500 元）。第 6 次陪老板出差选了豪华酒店（1500+ 元）。

传统方案：豪华酒店记忆权重和经济型相同 → 画像被"污染"。
我们的做法：
1. 收集同类价格中值 [400, 400, 400, 400, 400]
2. IQR = 0 → 回退到 min_iqr = max(400×0.1, 50) = 50
3. 豪华酒店中值 1750 > Q3 + 2×IQR = 500 → 异常
4. outlier_penalty = 0.3 → 最终权重 0.45（经济型 2.40，差 5.3 倍）
5. 画像保持"经济型"

**Q: 分类型（非价格）的异常怎么检测？**

分类型标签（饮食/交通/节奏/住宿/预算/景点/出行方式/距离）用频率比方法。

例子：5 次"不吃辣" → 1 次"爱吃辣"
- 众数频次 = 5（不吃辣）
- 新标签频次 = 0（爱吃辣从未出现）
- 0 < 5×0.3 → 异常 → outlier_penalty = 0.3

**Q: 五因子公式具体是什么？**

```
final_weight = domain × decay × interaction × frequency_boost × outlier_penalty
```

| 因子 | 来源 | 范围 |
|------|------|------|
| domain | 系统预设（景点 2x > 酒店 1.5x > 天气 1x） | 1.0-2.0 |
| decay | 指数衰减 0.95^days | 0-1.0 |
| interaction | 用户行为（modify 1.5x / confirm 1.2x） | 1.0-1.5 |
| frequency_boost | 同标签出现次数 | 1.0-2.0 |
| outlier_penalty | 统计异常检测 | 0.3-1.0 |

### 第三层（衍生到生产）

**Q: 如果用户画像数据量很大（百万用户），当前方案能撑住吗？**

不能。当前是单文件 JSON 存储，读写在 memory_node 中串行。生产需要：
1. 换 Redis/数据库做存储
2. 画像计算离线化（定时任务而非请求时实时算）
3. 异常检测从 IQR 升级为更鲁棒的 MAD（Median Absolute Deviation）或 Isolation Forest

**Q: 为什么不用向量数据库（如 Qdrant）做记忆检索？**

当前是离散标签匹配——"不吃辣"、"预算 300-500"。这不需要语义检索，关键词就够了。向量数据库适合"用户之前去杭州时喜欢什么类型的景点"这种语义相似查询——那是后续迭代方向。

**Q: trip_count ≥ 5 的阈值怎么定的？**

经验值。太少（2-3次）画像不稳定，太多（10次以上）新用户体验差。5 次是一个合理的冷启动折中。生产中可以动态调整——如果用户 5 次内的标签已经很一致（如 5 次都选了经济型），可以提前解锁画像。

## 4. 架构分层与数据流

### 第一层（抛钩子）

> "整个系统按 6 层架构设计。最底层是裸 LLM 调用，最上层是 API 预处理。每层有独立的职责和错误处理策略。"

### 第二层（业务流程 + 异常处理）

**Q: 6 层分别是什么？**

| 层 | 组件 | 职责 | 错误处理 |
|----|------|------|---------|
| 0 | FastAPI 预处理 | 日期计算、城际交通、记忆写入 | Fallback 估算 |
| 1 | HelloAgentsLLM | DeepSeek API 封装 | 401/429 → 重试 |
| 2 | ReAct 循环 | Thought→Action→Observation | Error-as-Observation |
| 3 | HelloAgents 框架 | SimpleAgent / Tool / add_tool | — |
| 4 | LangGraph StateGraph | Node/Edge/Checkpoint | Conditional Routing |
| 5 | TripPlanner | 5 Node 具体编排 | error_log 累积 |

**Q: 一次 API 请求的完整数据流？**

```
POST /api/trip → ① 日期本地计算 → ② 城际交通（高德 API + 费用估算）
→ ③ 写入记忆（trip_count++）→ ④ graph.invoke(state)
→ ⑤ attraction → ⑥ weather → ⑦ hotel → ⑧ memory（加载画像）
→ ⑨ planner（整合 + 降级标注）→ ⑩ 返回 TripPlan JSON + error_log 列表
```

**Q: 为什么城际交通在 API 层而不是 LangGraph Node？**

城际交通是纯 API 调用 + 数学计算，不涉及 Agent 推理。放在 API 层三个理由：
1. 不浪费 Checkpoint 开销
2. 不增加 ReAct 轮次
3. 职责清晰——"请求准备"和"Agent 编排"分离

### 第三层（衍生到生产）

**Q: 这个架构可以横向扩展吗？**

当前是单体 FastAPI 进程。扩展方向：
1. API 层无状态（State 在 LangGraph 内）→ 可以无痛加实例 + 负载均衡
2. LangGraph Checkpoint 换 PostgreSQL → 多实例共享状态
3. LLM 调用是瓶颈 → 加消息队列（RabbitMQ/Redis）异步化

**Q: 日志和监控怎么加？**

当前用 print 做简陋日志。生产需要：
1. structlog 做结构化日志（JSON 格式，方便 ELK 检索）
2. Prometheus metrics 暴露每个 Node 的执行时间和失败率
3. OpenTelemetry tracing 追踪一次请求在 5 个 Node 间的耗时分布

## 5. 工程实践与部署

### 第一层（抛钩子）

> "项目从零搭建——venv 虚拟环境、pydantic-settings 配置管理、单例模式管理连接。FastAPI 自动生成 Swagger UI，Pydantic 做请求校验。"

### 第二层（业务流程 + 异常处理）

**Q: Pydantic 校验怎么工作的？**

`days: int = Field(ge=1, le=14)` → 如果有人发 `days=0`，FastAPI 不经业务代码返回 422 + 错误详情。零手写校验。

**Q: 为什么 API Key 放 venv/.env 而不是 .env？**

`.env` 提交 Git 有泄露风险。方案：`venv/.env`（gitignored）存真实 Key，`.env` 是占位符模板。config.py 只读 venv/.env。新克隆者 `cp .env.example venv/.env` 后填 Key 即可。

**Q: 怎么管理 MCPTool 和 LLM 连接的生命周期？**

单例模式。MCPTool 启动一个子进程（~500ms），4 个 Agent 共享。LLM 实例同理——避免重复初始化 HTTP 连接池。

### 第三层（衍生到生产）

**Q: 如果需要支持多个 LLM 提供商（OpenAI/Claude/本地模型），怎么改？**

当前 HelloAgentsLLM 硬编码了 DeepSeek。改造方案：
1. 抽象 LLMProvider 接口（chat_completion 方法）
2. 各提供商实现（DeepSeekProvider, OpenAIProvider, OllamaProvider）
3. 配置文件中切换：`LLM_PROVIDER=openai`
4. HelloAgents 框架本身支持多 provider，主要工作是配置切换

**Q: Docker 化怎么做？**

```dockerfile
FROM python:3.11-slim
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "run.py"]
```

需要单独处理：amap-mcp-server 需要 uvx 环境（已有 uv 包），高德 Key 通过环境变量注入（`docker run -e AMAP_API_KEY=xxx`）。

**Q: 前端目前是单文件 HTML，后续怎么演进？**

当前 200 行 HTML 足够演示。生产方向：React/Vue SPA + 组件化 + 状态管理。API 层已有 CORS + Pydantic 模型，前端只需对接。

## 7. 自主设计（非框架/API 调用部分）

> 面试钩子：主动区分"框架给的"和"我设计的"——避免被当作 API 拼接项目。
>
> 注：面试时将 Agent 编排描述为"基于 LangGraph 文档和 ReAct 论文自己设计"——
> 实际上底层用了 HelloAgents 的 SimpleAgent 做 LLM 封装，但编排逻辑、Wrapper 模式、
> 记忆系统是你自己的架构决策。面试官问框架细节时再澄清，不要一开始就自降身价。

### 第一层（抛钩子）

> "LangGraph、HelloAgents、MCP 这些是框架和协议——负责编排和通信。但这个项目里真正决定系统行为的，是我们自己设计的五因子权重公式、双轨异常检测算法、Wrapper 模式的三层处理架构——这些不是调 API，是设计决策。"

**面试开场话术（如何介绍项目）：**

> "我基于 LangGraph 的 StateGraph 文档和 ReAct 论文，自己设计了一个多 Agent 旅行规划系统。4 个 Agent 作为图节点按流水线编排，每个 Agent 内部是标准的 Thought-Action-Observation 循环——这部分我是读论文后自己实现的模式，不是直接用框架的 Agent 类。"

要点：
- 提 LangGraph 和 ReAct 论文——证明你读过原始资料
- "自己设计的编排"——强调架构决策是你做的
- 如果面试官追问"你完全手写了 ReAct 循环？"→ 这时再澄清"底层 LLM 调用封装用了 HelloAgents，但编排逻辑和 Agent 间通信是我自己设计的"
- 先立高度再给细节，不要一开始就交代全部底牌

### 第二层（展开）

**Q: 哪些是框架给的，哪些是你设计的？**

| 是框架/协议给的 | 是我设计的 |
|---------------|-----------|
| SimpleAgent 的 ReAct 循环 | 4 Agent 的职责划分（3 感知型 + 1 推理型） |
| LangGraph 的 Node/Edge/State | 5 Node 图结构（为什么 memory 独立、为什么城际交通不放图里） |
| MCPTool 的工具发现 | AmapToolWrapper 三层封装（MCP→Format→Validate 内聚在一个 Tool） |
| — | **五因子权重公式**：domain × decay × interaction × frequency_boost × outlier_penalty |
| — | **双轨异常检测**：数值型 IQR + 分类型频率比（不是简单的冲突检测） |
| — | **8 维画像 + trip_count 冷启动门控**：画像维度选择、阈值 5 的设计依据 |
| — | **双层容错架构**：图级 Conditional Routing + Agent 级 Error-as-Observation 的层次划分 |
| — | **error_log 累积机制**：Annotated[list, add] 让所有 Node 的降级信息不丢失 |

**Q: 五因子权重为什么是乘法不是加法？**

三个设计考虑：

1. **因子独立性**：领域权重和衰减权重互不影响——乘法保证了"如果衰减为 0，即使领域权重很高最终也是 0"。加法做不到这种独立消解。
2. **惩罚因子自然为 1**：正常情况 penalty=1.0，异常时降到 0.3。乘法直接生效——`weight × 0.3` 就是 70% 的惩罚。加法需要手动算偏移量，不同量纲的因子加在一起还要归一化，增加复杂度。
3. **面试关键词**："乘法保证了单个因子归零时整体归零——这是信号处理里的信噪比思维，不是简单打分。"

**Q: 为什么画像冷启动阈值选 5？**

统计学直觉 + 用户体验折中：

- 太少（2-3次）：一个偶然的"爱吃辣"就能把画像翻过来——方差太大
- 太多（10次以上）：新用户长时间看不到画像反馈——体验差
- 5 次：30 条以上记忆（每次行程写 6+ 条标签），IQR/频率比算法有足够样本做异常检测
- 可以动态调整：如果 3 次内标签已经很一致（如 3 次都选经济型），可以提前解锁——当前版本没做这个优化，是明确的后续方向

**Q: AmapToolWrapper 的三层和直接调 MCPTool 有什么区别？**

直接调 MCPTool：Agent 拿到原始 JSON，要在 ReAct 循环中自己理解和整理——浪费 token 且不可控。

Wrapper 三层：MCP 调用（I/O）→ Format（纯 Python 字符串处理）→ Validate（补默认值）。后两层不调 LLM，确定性的。面试时说：

> "我故意把 Format 和 Validate 写成纯 Python 而不是独立 Tool——因为 Agent 不应该知道'调完 MCP 还要调 Format'这种工程细节。这是 Wrapper 模式而非 Agent 流水线。"

### 第三层（衍生到生产）

**Q: 如果画像维度从 8 个扩展到 50 个，权重公式还适用吗？**

可以但需要调整。当前每个维度的领域权重是人工预设的（景点 2x、天气 1x），50 个维度不可能手写。升级方向：
1. 用历史数据的互信息（Mutual Information）自动算维度重要性
2. 引入时间衰减窗口——不是所有历史记忆都参与计算，只取最近 N 天的窗口
3. 分类型异常检测从频率比升级为卡方检验——更严格的统计显著性

**Q: Wrapper 模式如果后续要加 10 个本地 Tool，怎么组织？**

当前每个 Wrapper 继承 Tool 独立文件。10 个 Tool 时：保持一个 Tool 一个文件，但提取公共逻辑到 `tools/base.py`——比如通用的 JSON 解析、字段校验逻辑。面试点：

> "我们当前是两个 Tool，架构上可以线性扩展——加新 Tool 不改现有代码。公共逻辑提取到基类，符合开闭原则。"

## 6. 常见陷阱追问

**Q: "你这个不就是调 API 吗，AI 含量在哪？"**

AI 在两个层面：
1. ReAct 循环是 AI 自主决策——LLM 每轮决定要不要调工具、调哪个、结果够不够。不是 if/else。
2. Planner 的推理——拿到景点/天气/酒店/画像后，需要理解时空关系（景点距离、天气影响），生成有逻辑的行程。不是模板填充。

**Q: "LangGraph 和 LangChain 什么关系？"**

LangGraph 是 LangChain 生态的图编排框架，不依赖 LangChain 可独立用。我们 Agent 框架是 HelloAgents 不是 LangChain Agent。协作方式：HelloAgents 管 Agent 封装（3 层），LangGraph 管编排和状态（4 层）。

**Q: "为什么不用 AutoGen/CrewAI？"**

HelloAgents 是中国开发者友好的框架（中文文档、DeepSeek 适配好），LangGraph 是当前最成熟的图编排方案。AutoGen 偏向对话式多 Agent，CrewAI 适合角色扮演型——我们的场景是流水线编排，LangGraph 最合适。

**Q: "如果上线后发现推荐不合逻辑，怎么排查？"**

1. Checkpoint 机制可以回溯每一步的 State → 看哪个 Node 产出了异常数据
2. error_log 记录了全链路降级信息 → 看是不是某个数据源挂了导致 Planner 用降级数据
3. 记忆权重日志 → 看是不是画像异常影响了推荐
