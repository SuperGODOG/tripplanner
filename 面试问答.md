# TripPlanner 面试问答

---

## 项目介绍（30 秒，埋钩子引导追问）

> "我做了一个多智能体旅行规划系统，端到端从输入城市到输出结构化行程，中间不需要人工介入。
>
> 编排层基于 LangGraph 自己设计了一个 5 Node 的 StateGraph——4 个 Agent 加一个记忆加载节点，图级 Conditional Routing 和 Agent 级 Error-as-Observation 两层容错各管各的，一套流程三种降级策略，任何一个数据源挂了都不影响最终输出。
>
> 工具链没有把 MCP 调用的格式化和校验写成独立 Tool——做了个 Wrapper 把三层处理封在一起，Agent 只看到一个接口，后两层纯 Python 不消耗 LLM token。FallbackTool 在三个数据源全挂的时候还能兜底。
>
> 记忆这边自研了一套五因子权重公式——领域权重、时间衰减、交互修正、频率加成、异常惩罚五个维度乘起来。异常检测分两轨——数值型价格走 IQR 统计分布，分类型偏好走频率比。八维用户画像冷启动五次才生效，偶发行为污染不了长期偏好。"

埋了五个钩子，每个后面都有对应回答：编排容错 → 话题 1，Wrapper 设计 → 话题 2，五因子权重 → 话题 3，双轨检测 → 话题 6，Fallback 兜底 → 话题 4。

## 偏业务面（面试官可能是 PM / 非技术主管）

### 1. 这个方案到底解决了现有方案没解决的什么问题？

**一句话回答**：传统平台需要用户手动在不同工具间切换，并且不理解"你这个人到底喜欢什么"——我们做到了端到端自动化和持续学习的用户理解。

**展开细节**：

一、手动切换 vs 一次输入

- **传统方案**：用携程查景点 → 用天气 App 查天气 → 用 Booking 查酒店 → 人脑拼成行程
- **我们的方案**：输入城市和日期，一次调 4 个 Agent 自动并行/串行拉取景点、天气、酒店，Planner 整合输出结构化 JSON 行程
- **核心差异**：不是信息的搬运工，是信息的编排者。携程告诉你"有什么"，我们帮你排"先干什么再干什么、怎么去、会不会下雨要不要调整"

二、传统记忆不区分偶然行为和长期偏好

- **传统方案**：用户每次搜索/点击都记下来，简单统计 → 一次陪老板出差选奢华酒店就会污染"经济型"画像
- **我们的方案**：五因子权重公式 + 双轨异常检测。Alex 5 次选经济型酒店（300-500 元），第 6 次选奢华酒店（1500+ 元）→ IQR 统计检测识别为异常，penalty = 0.3 → 最终权重 0.45 vs 经济型 2.40，差 5.3 倍 → 画像保持"经济型"

三、传统行程生成后不校验可行性

- **传统方案**：景点列表 + 酒店列表 = 扔给用户自己判断
- **我们的方案**：离群检测标出"八达岭距天安门 70km"这类异常景点，酒店回环校验确保不绕路；error_log 累积机制让 Planner 在整合时标注降级数据（"⚠️ 景点数据不可用，以下为通用推荐"）

---

### 2. 为什么不用现成的旅行 App？

**一句话回答**：我们不是替代携程，是"规划辅助"——携程告诉你有什么，我们帮你排顺序，而且越用越懂你。

**展开细节**：

一、定位差异

- 携程/飞猪：OTA（Online Travel Agency）——交易平台，卖票卖酒店，赚佣金
- TripPlanner：规划辅助工具——不卖任何东西，帮你把"去哪玩、什么时候去、怎么安排顺序"这件事自动化
- 类比：携程是"超市"，我们是"营养师"——超市有所有食材，营养师帮你配菜谱

二、跨会话记忆

- 携程每次搜索从零开始，不记得你上次选了经济型还是豪华型
- 我们的记忆系统跨会话保持：你在杭州选过经济型、不吃辣、偏好步行 → 下次规划北京行程时，这些偏好自动加权注入 Planner 的提示词，结果天然偏向你的习惯
- "越用越懂你"不是口号，是五因子权重持续更新的工程结果

三、整合多源数据的能力

- 携程不知道天气会不会下雨，不知道城际交通怎么走
- 我们同时拉景点、天气、酒店、交通四路数据，Planner 做语义理解层面的时空推理——"明天下雨，所以室外景点放上午，下午安排室内博物馆"

---

### 3. 如果上线，用户第一反应"这有什么用"怎么回答？

**一句话回答**：Demo 阶段核心展示 Agent 编排能力；实际价值在"自动整合多源数据"和"持续学习偏好"这两个点上。

**展开细节**：

一、Demo 阶段的价值展示

- 这不是一个成熟的 C 端产品，是一个 Agent 系统能力的 Demo
- 展示的核心不是"行程多完美"，而是：
  - 4 个 Agent 如何协作：谁负责什么、谁依赖谁、出错了怎么降级
  - 一个数据源挂了（比如高德酒店 API 限流），系统不崩溃，Planner 用降级数据继续出结果
  - 前端降级面板让用户看到"哪个数据源挂了、用了什么替代方案"

二、实际价值点 1：自动整合多源数据

- 用户只需要说"去北京 3 天"，剩下的景点搜索、天气查询、酒店推荐、交通计算全部自动完成
- 不需要打开 4 个 App 手动比较和拼凑
- 尤其适合"对目的地不熟、不想花时间做攻略"的场景

三、实际价值点 2：持续学习偏好

- 第一次用可能只是"通用推荐"，第五次开始，系统记得你"不吃辣、偏好经济型酒店、喜欢步行"——行程自动避开川菜馆、推 300 元酒店、优先步行可达的景点
- 这是传统旅行 App 做不到的——它们的"个性化"是标签过滤（选"经济型"筛酒店），我们的是行为学习（从你 5 次选择中推断偏好）

## 偏技术面（面试官是技术主管 / 架构师）

### 1. State 传递是全部数据还是经过节点提取的数据？

**一句话回答**：两者都有——attraction_data/weather_data/hotel_data 是 LLM 原始输出全文传递，center_lng/center_lat 是节点本地提取后注入的结构化数据。

**展开细节**：

一、全文传递的数据

- `attraction_data`、`weather_data`、`hotel_data`：LLM 生成的原始文本（景点列表、天气描述、酒店推荐）
- 为什么不提取？Planner 需要原始文本做语义理解——"长城适合上午去"这种推理需要自然语言上下文，结构化字段（景点名 + 坐标）不够
- 代价：State 体积较大，但当前场景（单次规划 4 个 Agent）在 LLM 上下文窗口内完全可控

二、结构化提取的数据

- `center_lng`、`center_lat`：Attraction Agent 从景点列表中计算的地理中心点
- 这是纯 Python 计算（对所有景点坐标求均值），不经过 LLM
- 注入 State 后，Hotel Agent 用它做附近酒店搜索，Planner 用它做离群检测
- 设计原则：**能确定计算的不要让 LLM 做，能本地提取的不要让下游重复算**

三、为什么这样设计

- 全文给 Planner 做"理解"，结构化数据给下游做"校验"——两种需求不同，不能互相替代
- 如果把所有数据都提前提取成结构化字段，Planner 失去了语义上下文，生成质量下降
- 如果只传全文不给结构化数据，离群检测、酒店回环等校验逻辑无法运行

### 2. 如何保证 State 不丢失关键信息？

**一句话回答**：`Annotated[list, add]` 保证 error_log 累积而非覆盖，每个节点只返回 partial dict 由 LangGraph 自动 merge，重试计数器防止死循环。

**展开细节**：

一、Annotated[list, add] 累积机制

```python
error_log: Annotated[list, add]
```

- 普通 dict 赋值 `state["error_log"] = [...]` 会覆盖前一个节点写入的内容
- `Annotated[list, add]` 告诉 LangGraph：每次写入不是替换，是追加（相当于 `extend`）
- 效果：4 个 Node 各写各的降级日志，最终 State 里是全链路累积的完整日志
- 前端降级面板直接读取这个列表，展示每一步的降级情况

二、Partial Dict 自动 Merge

- 每个 Node 只返回自己负责的字段，不返回完整 State
- 例如：`attraction_node` 返回 `{"attraction_data": ..., "center_lng": ..., "center_lat": ...}`
- LangGraph 框架自动将 partial dict merge 到当前 State，不覆盖其他字段
- 好处：Node 之间解耦——hotel_node 不需要知道 attraction_node 写了哪些字段

三、重试计数器防止死循环

- `planner_retry_count`、`hotel_retry_count`：在 State 中维护
- Conditional edge 逻辑：
  ```python
  if planner_retry_count >= 2:
      return "done"  # 超过上限，强制结束
  ```
- 防止"酒店不匹配 → 重试 → 还是不对 → 无限循环"的死锁情况

### 3. 降级信息是后端内部用的还是发给下一个 Node 的？

**一句话回答**：error_log 同时做两件事——后端内部日志记录 + 前端降级面板展示，通过软伤/硬伤分类和状态标记让下游 Node 知道自己拿到的是降级数据。

**展开细节**：

一、软伤（warnings）

- 定义：数据源部分失败但仍有可用数据（如天气 API 超时但景点数据正常）
- 行为：写 error_log，不阻断流程，Node 继续执行
- 前端展示：降级面板显示黄色警告（"⚠️ 天气数据不可用，使用通用穿衣建议"）
- 下游感知：status 标记为 "degraded"，Planner 在提示词中加入降级标注

二、硬伤（hard_errors）

- 定义：数据源完全失败且影响下游（如酒店 API 全部挂掉）
- 行为：写 error_log + 触发 conditional edge 回到对应 Node 重试
- 重试逻辑：
  ```python
  graph.add_conditional_edges("planner", check_status, {
      "retry_hotel": "hotel",
      "done": END
  })
  ```
- 上限：retry_count >= 2 后强制结束，用 FallbackTool 兜底

三、状态标记（status 字段）

- `attraction_status`、`weather_status`、`hotel_status`：每个 Node 写入自己的执行状态
- 可选值：`"success"` / `"degraded"` / `"failed"`
- `attraction_status = "failed"`：Planner 看到后知道景点数据不可用 → 不在行程中放具体景点，而是写降级文案
- 关键设计：**状态标记让下游 Node 有条件判断能力，而不是盲信数据**

### 4. 为什么天气不放在图里而是 API 层？

**一句话回答**：天气是纯查询——不依赖 Agent 数据、不需要 Checkpoint、不需要 LLM，和城际交通一样属于"请求预处理"而非"Agent 编排"。

**展开细节**：

一、天气为什么不需要进图

- 天气查询是确定性的：输入城市 + 日期 → 输出温度、天气状况 → 纯 API 调用
- 不需要 Agent 做决策：不像"搜索景点"需要 LLM 理解用户偏好选择关键词
- 不需要 Checkpoint 恢复：天气查一次就够了，中间失败重试即可，不需要从断点恢复
- 不需要 ReAct 循环：没有"观察结果后决定下一步"的逻辑

二、城际交通同理

- 城际交通 = 高德路径规划 API + 费用估算公式 → 纯计算
- 如果放进图里作为 Node：增加 Checkpoint 开销、增加 State 序列化体积、增加一次 ReAct 轮次
- 放在 API 层（FastAPI 预处理）三个好处：
  1. 不浪费 Checkpoint 开销（每次请求自动快照，串行化大体积 State 有成本）
  2. 不增加 ReAct 轮次（每个 LLM 调用都是 token 消耗）
  3. 职责清晰——"请求准备"和"Agent 编排"分离

三、失败处理

- API 层失败时直接写降级文本注入 State（如"城际交通数据不可用，请自行查询"），不消耗 ReAct 轮次
- 和图中 Node 失败的差异：图中失败需要走 conditional edge 逻辑（有 LLM 参与判断），API 层失败在 Python 代码中确定性处理

### 5. 离群检测为什么用标准差法而不直接用距离阈值？

**一句话回答**：固定阈值（如 10km）无法适用于不同规模的城市——北京 sigma 大所以阈值大（只标八达岭），成都 sigma 小所以阈值小（更敏感）。

**展开细节**：

一、固定阈值的致命缺陷

- 如果设硬阈值 10km：
  - 北京：六环内大部分景点都在 10km 以外 → 疯狂标记"异常"，误报率极高
  - 成都：市区景点集中在 5km 内 → 从来不会触发，漏掉真正的异常
- 固定阈值 = 一个尺子量所有城市 → 只对某一种城市规模有效

二、标准差法的自适应原理

```python
threshold = min(mean + 2 * sigma, 80)  # 80km 硬上限兜底
```

- 计算所有景点到城市中心距离的均值和标准差
- 北京：sigma 大（景点分散）→ 阈值大 → 只标八达岭（真正远离市区的）
- 成都：sigma 小（景点集中）→ 阈值小 → 更敏感，稍微偏远的都能检出
- 本质：用数据自身的分布特征决定判定标准，而非拍脑袋的魔数

三、80km 硬上限兜底

- 为什么加？极端情况：重庆市域东西跨度 400km+，sigma 可能极大
- 如果完全信任标准差，巫山到渝中 200km 可能不会被标为异常
- 80km 硬上限确保：不管什么城市，超过 80km 一定标为异常
- 面试关键词："自适应统计 + 工程兜底，两重保险"

## 常见陷阱追问

**Q: "你这个不就是调 API 吗，AI 含量在哪？"**

AI 在两个层面：
1. ReAct 循环是 AI 自主决策——LLM 每轮决定要不要调工具、调哪个、结果够不够。不是 if/else。
2. Planner 的推理——拿到景点/天气/酒店/画像后，需要理解时空关系（景点距离、天气影响），生成有逻辑的行程。不是模板填充。

**Q: "LangGraph 和 LangChain 什么关系？"**

LangGraph 是 LangChain 生态的图编排框架，不依赖 LangChain 可独立用。我们 Agent 框架是 HelloAgents 不是 LangChain Agent。协作方式：HelloAgents 管 Agent 封装（3 层），LangGraph 管编排和状态（4 层）。

**Q: "为什么不用 AutoGen/CrewAI？"**

HelloAgents 是中国开发者友好的框架（中文文档、DeepSeek 适配好），LangGraph 是当前最成熟的图编排方案。AutoGen 偏向对话式多 Agent，CrewAI 适合角色扮演型——我们的场景是流水线编排，LangGraph 最合适。

**Q: "如果上线后发现推荐不合逻辑，怎么排查？"**

1. Checkpoint 机制可以回溯每一步的 State → 看哪个 Node 产出了异常数据
2. error_log 记录了全链路降级信息 → 看是不是某个数据源挂了导致 Planner 用降级数据
3. 记忆权重日志 → 看是不是画像异常影响了推荐
